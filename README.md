# 1. このリポジトリの目的

このリポジトリは、以下の技術思想を体系化した **外側OS（Outer Structure OS）** の原典です。

- 文法（10アクション）を外側に固定する  
- ロジック（core）を黒箱化する  
- public → model → core の3層構造  
- 文法とロジックの完全分離  
- AI時代に最適化されたプログラミング構造  

> 自然言語の文法が少なくても無限の表現ができるように、  
> この10アクションも“文法”なので、組み合わせれば無限の機能が作れる。

---

 2. 外側OS（Outer Structure OS）の構造

外側OSは以下の **3層構造** で成立する。

 ● parts / flow（外側の文法）
アプリケーションの文法を外側に固定する。  
ロジックは書かない。

 ● model（受け皿）
外側の文法を受け取り、core に渡す。

 ● core（ロジック・黒箱）
実際の処理を行う部分。  
外側の文法とは完全に分離されている。

---

 3. 黒箱アーキテクチャ（Black Box Architecture）

外側OSの中核となる概念。

- 内部ロジック（core）は外部から見えない  
- public と server を完全に分離  
- 外側は文法だけ  
- 内側はロジックだけ  
- 外側から内部を改変できない  

安全性・権利保護・市場性を支える基盤。

---

## 4. 文法（10アクション）

外側OSは、アプリケーションの操作を以下の10アクションに固定する。

query        : データ取得

filter       : 条件絞り込み

sort         : 並び替え

render       : 表示・出力

authenticate : 認証

save         : 新規保存

update       : 更新

delete       : 削除

route        : 画面遷移

notify       : 通知


これらは「外側の文法」であり、  
ロジック（core）はこれらのアクションを通じてのみ実行される。

自然言語の文法が少なくても無限の表現ができるように、  
この10アクションも“文法”であり、  
組み合わせることで無限の機能を構築できる。

プラモデル方式プログラミング思想 — 外側OS・文法規格・黒箱構造の原典（S）

本思想および外側OS・文法規格・黒箱構造は、 S（しょう）によって提案・定義・体系化されたものです。 本ドキュメントはその正式な原典です。

 1.この思想が生まれた背景
現代のプログラミングは「コードを書くこと」が前提になっている。  
しかし実際には、

- 初心者は文法で挫折する  
- 経験者は同じコードを何度も書く  
- 市場はテンプレが乱立する  
- AIはコードを大量に生成し始める  
- 言語の壁は依然として存在する  

この状況は **構造的に限界** を迎えている。

そこで私は、  
**プログラミングを“書くもの”から“組み立てるもの”へ変える思想**  
を考案した。

---

 2.プラモデル方式とは何か（思想の定義）

プラモデル方式とは、  
**プログラムを“部品（ロジック）”として作り、  
それを“設定”と“組み合わせ”で動かす思想**  
である。

コードを書くことは目的ではなく、  
**部品を組み立てるための手段** になる。

---

 3.この思想が解決する問題

 ■ 初心者の挫折
- 文法・エラー・環境構築で止まる  
- 動くものが見えない  

 ■ 経験者の非効率
- 同じ処理を何度も書く  
- 言語ごとに思考を切り替える必要がある  

 ■ 市場の限界
- テンプレは“完成品”であり拡張性が低い  
- ロジックの著作権が守られない  

 ■ AI時代の矛盾
- AIがコードを書くのに、人間は“書く前提”の学習を続けている  

プラモデル方式は、これらを **構造から解決する思想** である。

---

 4.思想の構造（コア）

 ■ 4-1. 部品化（ロジックの最小単位化）
プログラムを「部品」として切り出す。  
部品は単機能で、再利用可能で、壊れにくい。

 ■ 4-2. 設定化（コードを書かずに動作を変える）
部品は設定ファイルで動作を変えられる。  
初心者は設定だけ触ればよい。

 ■ 4-3. 組み合わせ（部品を並べて機能を作る）
複雑な機能は部品を組み合わせることで生まれる。  
プラモデルと同じ発想。

 ■ 4-4. 権利構造（作者が守られる）
- ロジックの著作権は作者  
- ユーザーは“使う権利”だけ  
- 閲覧権は別  
- 改造は許可制  

 ■ 4-5. 市場構造（部品が売買される）
部品が増えるほど価値が増える。  
App Store と同じネットワーク効果が働く。

 ■ 4-6. 言語非依存
PHPでも、JavaScriptでも、Pythonでも成立する。  
思想は技術を超える。

 ■ 4-7. AI時代との一致
AIがコードを書く時代において、  
人間は「理解・判断・組み合わせ」を担当する。

---

 5.初心者が育つ理由（学習モデル）

- いきなり動く  
- 設定だけ触る  
- 壊れない  
- 触る → 変わる → 理解する  
- 必要なときだけコードを読む  

これは **従来の学習モデルの逆転** である。

---

 6.経験者が伸びる理由（拡張モデル）

- 部品を追加できる  
- 改造できる（許可制）  
- 言語を超えて同じ思想で開発できる  
- 自分の部品を市場に出せる  

---

 7.市場の成立（経済モデル）

部品市場は自然に成立する。

- 基本部品  
- 拡張部品  
- 高度部品  
- 閲覧権  
- テンプレ  
- 学習コンテンツ  

これらが売買されることで、  
**新しいプログラミング経済圏** が生まれる。

---

 8.この思想が世界に広がる理由（構造的優位性）

- 初心者が育つ  
- 経験者が得をする  
- 言語に依存しない  
- 市場が自動で広がる  
- 権利が守られる  
- AI時代に最適  
- エンジニアの思考を構造化している  

これは **技術ではなく“新しい産業の構造”** である。
