# Compatibility (互換性 / 組み合わせ)

PMP（php-plamodel-method）は「外側固定 × 文法10アクション × ロジック一本化」により、
既存のフレームワーク・ローコードツール・API構造と競合せずに共存できます。
外側OSはロジックを持たず、UI・文法・ロジックを完全分離するため、
他技術を“内側の部品”として安全に取り込むことができます。

---

## Frameworks（既存フレームワークとの相性）

外側OSはロジックを持たないため、Laravel / Symfony / CakePHP / WordPress など
既存フレームワークと競合せずに組み合わせ可能です。

- parts → フレームワークのビューやコンポーネントを利用可能  
- model → フレームワークの機能を文法10アクションに翻訳  
- core → フレームワークのロジックを黒箱化して安全に統合  

既存FWを「内側の部品」として扱えるため、  
外側OSは FW の自由度と OS の安全性を同時に得られます。

---

## Low-code / No-code（ローコード・ノーコードとの相性）

UI層をローコードツール（Bubble / Retool / AppSheet）に任せ、
文法層・ロジック層を外側OSで固定する構造が成立します。

- UIはローコードで高速開発  
- 文法10アクションで外側の統一  
- core（黒箱）でロジック一本化  

ローコードの弱点である「ロジックの散乱」を  
外側OSが完全に解消します。

---

## API / Microservices（API・マイクロサービスとの相性）

core.php が唯一のロジック実装場所であるため、
API化・マイクロサービス化が容易です。

- parts → API呼び出し  
- model → APIレスポンスを文法10アクションに変換  
- core → APIロジックを黒箱として統合  

外側OSは「ロジックを外に出さない」ため、  
APIや外部サービスとの境界が常に安全に保たれます。

---

## Security（セキュリティ面の相性）

依存ゼロ構造により攻撃面が最小化され、
ロジック一本化によりセキュリティ管理が単純化されます。

- 外側は parts/model のみ  
- 内側は core（黒箱）だけ  
- 依存ライブラリがないため脆弱性が発生しにくい  

外側OSは「攻撃面の縮小」を構造的に実現します。

---

## AI-generated Code（AI生成コードとの相性）

parts/model の生成をAIに任せ、
core（黒箱）を人間が管理する構造が成立します。

- AIが UI と文法層を生成  
- 人間が core を守る  
- 文法10アクションで構造が崩れない  

AI時代において、  
外側OSは「AIが壊せない構造」を提供します。

---

## Summary（まとめ）

PMP の外側OSは、既存技術を“内側の部品”として安全に取り込むための構造です。

- 競合しない  
- 統合しやすい  
- セキュリティが強い  
- AI時代に適応している  

これらの特性により、  
PMP は既存技術との互換性が非常に高く、  
外側OSとしての強みを最大限に発揮します。
